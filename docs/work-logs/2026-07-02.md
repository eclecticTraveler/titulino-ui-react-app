## What was done

### T34 тАФ DB: composite type + rewritten claim function
**File:** `titulino-warehouse/deploy/Lrn/2026/07/02_know_me_ai_bilingual_feedback.sql`

Created composite type `"Lrn".claimed_know_me_ai_job` тАФ mirrors all 14 columns of `KnowMeAiJob` plus `"NativeLanguageId" text`. Rewrote `"Lrn".claim_next_pending_know_me_ai_job()` to return `SETOF "Lrn".claimed_know_me_ai_job` using a CTE + `LEFT JOIN "Enrollment"."Course"` on `CourseCodeId` to derive `NativeLanguageId` at claim time. LEFT JOIN ensures claim succeeds even if course has no language set.

Updated `"TitulinoApi_v1"."ClaimNextPendingKnowMeAiJob"()` wrapper: `v_job` type changed to `"Lrn".claimed_know_me_ai_job`, `NativeLanguageId` added to jsonb output.

**Deploy fix:** The initial `sqitch deploy` failed with `cannot change return type of existing function`. Fixed by adding `DROP FUNCTION IF EXISTS "Lrn".claim_next_pending_know_me_ai_job();` before the `CREATE FUNCTION` in the migration. `CREATE OR REPLACE` cannot change a function's return type in PostgreSQL тАФ a drop is required.

### T35 тАФ DB: GetKnowMeAiJobResult returns OriginalEssays
Same migration file. Added `'OriginalEssays', v_job."EssaysJson"` to the `jsonb_build_object` in `"TitulinoApi_v1"."GetKnowMeAiJobResult"`. No inner function change needed тАФ it already returns the full `KnowMeAiJob` row.

- `IKnowMeAiJob.cs`: added `string? NativeLanguageId { get; }`

### T37 тАФ Worker: NativeLanguageId threaded to EssayInput
Added `string? NativeLanguageId` to `EssayInput.cs`. `KnowMeAiManager.ProcessJobAsync` sets it from `job.NativeLanguageId` when building inputs. No change to `CorrectEssaysAsync` signature тАФ the value rides per-essay through `EssayInput`.

### T38 тАФ Worker: bilingual prompt + plain-text rule
`AiPromptHelper.BuildEssayCorrectionPrompt` gains optional `string? nativeLanguageId` parameter. When non-null, the prompt adds three native fields to the JSON template (`nativeSummaryFeedback`, `nativeGrammarNotes`, `nativeVocabularySuggestions`) and instructs the AI to translate the feedback into that language. Added explicit plain-text rule: "no asterisks, no bold, no italic, no markdown formatting of any kind" тАФ this fixes the `**markdown**` visible in current production grammar notes.

### T39 тАФ Worker: EssayResult native fields + all three providers
`EssayResult.cs` gains `NativeSummaryFeedback`, `NativeGrammarNotes`, `NativeVocabularySuggestions`. All three providers (`GeminiAiProvider`, `OpenAiProvider`, `AnthropicAiProvider`) updated to:
- Pass `essay.NativeLanguageId` to `AiPromptHelper.BuildEssayCorrectionPrompt`
- Parse `nativeSummaryFeedback`, `nativeGrammarNotes`, `nativeVocabularySuggestions` from the AI JSON response

### T40 тАФ Worker: native feedback in FeedbackJson
`KnowMeAiManager.ProcessJobAsync` `feedbackDict` now includes `nativeSummary`, `nativeGrammarNotes`, `nativeVocabularySuggestions` per question. These land in `FeedbackJson` in the DB alongside the existing English fields.

**Implementation note:** The plan described a nested `feedbackNative` object. The actual implementation uses flat keys (`nativeSummary`, `nativeGrammarNotes`, `nativeVocabularySuggestions`) at the same level as the English keys. This was simpler to parse in both the AI response and the UI without introducing a nested type.

### T41 тАФ .NET API: OriginalEssays in DTO
- `KnowMeAiResultResponse.cs`: added `public JsonElement? OriginalEssays { get; set; }`
