# 2026-06-29 — Audience Location UX + Division Support Planning

## Domain: Feature — Planning only, no code changes

### Context

New session. User brought two questions about the audience segmentation location filter:
1. The Dev UI already has separate Residency/Birth rows (from the 2026-06-26 plan), but has no visual indication that leaving both blank = "All Locations". User asked whether that is correct and whether a UI indicator is possible.
2. User wants country division dropdowns to show only divisions where students are actually enrolled (not all geographic divisions), with "Unknown" for students who have a country but no division on record.

Full exploration of the codebase and backend was done before writing the plan.

### Findings

- **Dev UI**: Dual-row location filter (Residency + Birth) is already implemented in `GlobalAdminToolsLandingDashboard.js`. LOB default filters and payload builder in `AudienceMessaging.js` also have the new fields (`residencyCountry`, `birthCountry`, etc.) alongside the old ones.
- **Data flow confirmed**: React app calls Supabase RPC **directly** at `https://...supabase.co/rest/v1/rpc/GetContactSegment`. The `titulino-communication` C# library is used by the worker/email service, not by the React UI. No C# changes are needed for the segmentation UI.
- **Supabase stored procedures**: `GetContactSegment` and `GetCountryDivisions` still use the old parameters. These are managed directly in Supabase.
- **Country divisions**: `GetCountryDivisions` returns all geographic divisions for a country, not filtered by enrollment. The call also does not pass `locationType` correctly per row (always sends `"all"`).
- **Correction**: Initial plan draft incorrectly included C# backend tracks (B1–B3). These were removed after confirming the React UI does not go through the C# layer for segmentation calls.
