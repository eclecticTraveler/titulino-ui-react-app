### Bug fix — `get_country_divisions` ORDER BY runtime error (0A000)

**Symptom**: After deploying C1, selecting a country in the Residency or Birth dropdown showed "No hay datos" in the Region dropdown. Browser network tab showed HTTP 400 from `GetCountryDivisions` with:
```json
{ "code": "0A000", "message": "invalid UNION/INTERSECT/EXCEPT ORDER BY clause",
  "details": "Only result column names can be used, not expressions or functions." }
```

**Root cause**: PostgreSQL does not allow expressions or function calls in the `ORDER BY` of a bare `UNION ALL`. Our C1 migration used `ORDER BY CASE WHEN "CountryDivisionName" = '__unknown__' THEN 1 ELSE 0 END` directly on the `UNION ALL`, which is only legal in a regular `SELECT`. The function compiled successfully (plpgsql defers query parsing to runtime) so no error appeared at deploy time.

**Fix**: New migration `Enrollment/2026/06/29_fix_get_country_divisions_order_by` wraps the entire `UNION ALL` in a subquery alias (`cdiv`). The outer `SELECT cdiv."CountryDivisionName" FROM (...) cdiv ORDER BY CASE WHEN ...` is a regular SELECT, where expressions in `ORDER BY` are permitted. Logic is identical.

New sqitch files:
- `deploy/Enrollment/2026/06/29_fix_get_country_divisions_order_by.sql`
- `revert/Enrollment/2026/06/29_fix_get_country_divisions_order_by.sql`
- `verify/Enrollment/2026/06/29_fix_get_country_divisions_order_by.sql`

Deployed and confirmed working — Region dropdown now populates after selecting a country.

**Key lesson**: `CREATE OR REPLACE FUNCTION` in plpgsql does not validate inner query syntax at creation time for `RETURN QUERY` statements. Always test the function call, not just the deploy, when adding UNION/ORDER BY patterns.

## Domain: Know Me AI Essay Review — Architecture + Planning

### Context

New feature: students write essay answers in `KnowMeV3`, AI corrects each essay, corrections displayed back in the same component.

### Key finding — architectural problem

The existing `KnowMeAiManager` in `TitulinoWorkerService` polls a GCS bucket for `/ai-inbox/` files. But submissions go through `UpsertAuthenticatedKnowMeSubmission` Supabase RPC → PostgreSQL. Nothing writes to the GCS inbox. The worker would idle forever. `IRepositoryClient` is injected into the manager but completely unused.

### Architecture decision

DB-driven job queue. `UpsertAuthenticatedKnowMeSubmission` will also insert a `pending` row into a new `Lrn.KnowMeAiJob` table. Worker polls the DB via `IRepositoryClient` (already wired, not yet used). GitHub CI/CD for worker already exists (`deploy-titulino-worker.yml`).

### Key implementation notes

- **Claim atomicity**: `FOR UPDATE SKIP LOCKED` in `claim_next_pending_know_me_ai_job` — safe for concurrent worker instances. Get+claim are one operation (no separate GetPending RPC needed).
- **Essay filtering**: `ensure_know_me_ai_job` uses `jsonb_typeof(value) = 'string'` to extract only essay answers from `Answers` jsonb, skipping photo arrays and consent booleans. If no string answers exist, the function returns early with no INSERT.
- **Re-submit safety**: `ON CONFLICT ("SubmissionId") DO NOTHING` — a student re-submitting doesn't reset an in-progress or completed AI job.
- **Fault isolation**: AI job creation wrapped in its own `BEGIN…EXCEPTION` block inside the upsert loop. A failure logs `WARNING` via `RAISE NOTICE` but never breaks the student's submission response.
- **Worker RPCs grant only service_role** — `ClaimNextPendingKnowMeAiJob`, `CompleteKnowMeAiJob`, `FailKnowMeAiJob` are not exposed to user roles.
- **Result RPC grants titulino_user** — `GetKnowMeAiJobResult` is called by the .NET API on behalf of the authenticated student.
