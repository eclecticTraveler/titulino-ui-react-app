### Bug fix ‚Äî `get_country_divisions` ORDER BY runtime error (0A000)

**Symptom**: After deploying C1, selecting a country in the Residency or Birth dropdown showed "No hay datos" in the Region dropdown. Browser network tab showed HTTP 400 from `GetCountryDivisions` with:
```json
{ "code": "0A000", "message": "invalid UNION/INTERSECT/EXCEPT ORDER BY clause",
  "details": "Only result column names can be used, not expressions or functions." }
```

**Root cause**: PostgreSQL does not allow expressions or function calls in the `ORDER BY` of a bare `UNION ALL`. Our C1 migration used `ORDER BY CASE WHEN "CountryDivisionName" = '__unknown__' THEN 1 ELSE 0 END` directly on the `UNION ALL`, which is only legal in a regular `SELECT`. The function compiled successfully (plpgsql defers query parsing to runtime) so no error appeared at deploy time.

**Fix**: New migration `Enrollment/2026/06/29_fix_get_country_divisions_order_by` wraps the entire `UNION ALL` in a subquery alias (`cdiv`). The outer `SELECT cdiv."CountryDivisionName" FROM (...) cdiv ORDER BY CASE WHEN ...` is a regular SELECT, where expressions in `ORDER BY` are permitted. Logic is identical.

New sqitch files:
- `deploy/Enrollment/2026/06/29_fix_get_country_divisions_order_by.sql`
- `revert/Enrollment/2026/06/29_fix_get_country_divisions_order_by.sql`
- `verify/Enrollment/2026/06/29_fix_get_country_divisions_order_by.sql`

Deployed and confirmed working ‚Äî Region dropdown now populates after selecting a country.

**Key lesson**: `CREATE OR REPLACE FUNCTION` in plpgsql does not validate inner query syntax at creation time for `RETURN QUERY` statements. Always test the function call, not just the deploy, when adding UNION/ORDER BY patterns.
