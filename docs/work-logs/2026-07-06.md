# Work Log ‚Äî 2026-07-06

## Session goal

Fix enrollment profile picture bypass: enrollees were submitting without photos because the Ant Design validator was never registered in early component states. Also harden upload failure handling and improve signed URL reliability.

---

## What was done

### T1 ‚Äî `EnrollmentProfilePictureField.js` ‚Äî Root cause fix (validator registration gap)

**Problem:** The component returned `null` in initial state and a loading `<Card>` (without `<Form.Item>`) during API lookup. Ant Design's `form.validateFields()` only runs validators for mounted `<Form.Item>` components ‚Äî if the item isn't in the DOM, the photo field is silently skipped and `filesMap.profilePictureUpload` is always empty.

**Fix:** Removed the two early-return paths that skipped rendering `<Form.Item>`. The card now always renders (with a `<Spin>` while checking, hidden `<Form.Item>` via `display:none`). The validator is always registered when `isEnabled=true`, so `validateFields()` correctly blocks submission until a photo is selected.

- Added `Spin` to antd imports
- `const isChecking = requirement.loading || !requirement.checked` ‚Äî controls visible/hidden state
- `style={isChecking ? { display: "none" } : undefined}` on `Form.Item` keeps it mounted but invisible
- Covers all 3 paths: authenticated, unauthenticated-found, brand-new

### T2 ‚Äî `QuickToFullEnrollment.js` + `AuthenticatedQuickEnrollment.js` ‚Äî Upload failure handling

**Problem (Quick):** When photo upload failed after enrollment, the code showed a dismissible `message.warning` and then opened the success modal anyway. User saw "enrolled!" even though their photo wasn't saved.

**Problem (Authenticated):** `uploadedProfilePicture` in the result was never checked ‚Äî the success modal always opened regardless.

**Fix:** Both components now:
- Check `uploadedProfilePicture?.wasUploaded` after enrollment
- If false: reset the photo field (`setFieldsValue({ [PROFILE_PICTURE_FIELD_NAME]: [] })`), set `photoUploadError=true`, and return early (no modal)
- Render a warning `<Alert>` above the submit button telling the user to re-upload and resubmit
- Enrollment is idempotent ‚Äî resubmitting updates the existing record and retries the photo

### T4 ‚Äî `LrnManager.js` ‚Äî Production error logging for upload failures

Added `console.error` in `ensureEnrollmentProfilePicture` at the two silent failure points:
1. `!resolvedContext.contactInternalId || !resolvedContext.token` ‚Äî context could not be resolved
2. `!result.wasUploaded` ‚Äî upload call returned no profileUrl

These fire in all environments (prod included), unlike `logEnrollmentProfilePictureDebug` which is non-prod only.

### T5 ‚Äî `KnowMeProfiles.js` + `TitulinoNetService.js` ‚Äî Signed URL TTL increase

Changed `DEFAULT_PROFILE_URL_AVAILABILITY_MINUTES` from 128 ‚Üí 480 (8 hours) in `KnowMeProfiles.js`.

Also updated the hardcoded `minutesRequestedForProfileUrlUsage: 128` ‚Üí `480` in the single-profile lookup request body in `TitulinoNetService.js` (line 469).

**Not done (follow-up):** `onError` URL-refresh handler on grid `<Image>` components. The profile images are rendered via `renderAvatarCell()` in LOB files (`AdminInsights.js`, `ShopAnalytics.js`). Adding a stateful re-fetch requires changing the `renderAvatarCell` signature to accept an `onError` callback, then wiring it in the column definitions. Noted as a follow-up task.
