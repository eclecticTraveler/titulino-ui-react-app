### What T30–T33 actually are

The foundation (structured JSON output, per-essay loop, JSON parsing) was already built in T13–T15. Phase 5 upgrades the *schema* from a flat `{ correctedText, summaryFeedback }` to a richer structure with arrays.

```json
{
  "questionId": "q2_bio",
  "correctedText": "...",
  "summaryFeedback": "...",
  "grammarNotes": ["..."],
  "vocabularySuggestions": ["..."]
}
```
Update `EssayResult.cs` to add `GrammarNotes` and `VocabularySuggestions` (both `List<string>`).

**T31** — Grammar topic injection: `EssayInput.GrammarTopic` is currently always empty. To inject the real topic, the worker needs a `questionId → grammarTopic` mapping. This mapping is in `know-me-survey-data.json` in the UI. Two options for the session:
- **Option A (simpler, no DB change):** Put a static `Dictionary<string, string>` in the worker (e.g. `"q2_bio" → "Descriptive Writing"`) — hard-coded until the DB stores it.
- **Option B (clean):** Add `grammar_topic` to the `EssaysJson` structure when the job is created (small DB change + Sqitch migration).
  
Decide at the start of the session.

**T32** — Update the `AnthropicAiProvider.TryParseResult` to extract the two new array fields. Also update `OpenAiProvider.ParseResult` and `GeminiAiProvider.ParseResult` with the same change.

**T33** — Update `KnowMeAiManager.ProcessJobAsync`:
- `correctedEssaysJson` DB column stores `{ questionId: correctedText }` — unchanged
- `feedbackJson` DB column should now store the richer per-question object instead of just the summary text
- Or keep the flat feedback string and add a separate `grammarNotesJson` column (T30 decision)

### Key decision before writing code

The DB columns `corrected_essays_json` and `feedback_json` currently store flat `{ questionId: "text" }`. If T30 adds `grammarNotes` and `vocabularySuggestions`, decide at the start whether to:
1. Store the richer object in `feedback_json` directly (no migration — UI just changes how it reads the feedback)
2. Add a new `ai_details_json` column (requires a Sqitch migration)

Option 1 is fastest and doesn't require a DB change.
