## Domain: Redux Middleware Architecture — Why thunk is disabled

### Context

During the package audit, `redux-thunk` was flagged as unused. The store explicitly disables it. Documented here for future reference so any agent or developer understands the async middleware choice and does not re-enable thunk by accident.

### How the store middleware is configured

File: `src/redux/store/index.js`

```js
const customMiddlewares = [
  sagaMiddleware,
  reduxPromise,
  axiosMiddleware(HttpService.getAxiosClient()),
];

const store = configureStore({
  reducer: reducers,
  middleware: (getDefaultMiddleware) =>
    getDefaultMiddleware({ thunk: false, serializableCheck: false, immutableCheck: false })
      .concat(customMiddlewares),
});
```

RTK's `configureStore` includes `redux-thunk` by default. `{ thunk: false }` removes it from the defaults before the custom middleware is appended.

### What thunk does (and why this app doesn't need it)

Thunk lets an action creator return a **function** instead of a plain object. That function receives `dispatch` and does async work before dispatching the real action:

```js
// Thunk pattern — NOT used here
const loadUser = () => async (dispatch) => {
  const data = await api.getUser();
  dispatch({ type: 'SET_USER', payload: data });
};
```

### What this app uses instead

| Middleware | Purpose |
|---|---|
| `redux-promise` | Action creators return a plain object with a **Promise as the payload**. The middleware waits for the Promise to resolve and dispatches the result automatically. |
| `redux-saga` | Handles complex async side effects (auth flows: SIGNIN, SIGNOUT, SIGNUP) via generator functions in `src/redux/sagas/`. |
| `redux-axios-middleware` | Wraps axios calls dispatched as actions — automatically handles request/success/error lifecycle. |

The app's async action creator pattern (using `redux-promise`):

```js
// Promise pattern — what this app uses
export const getCourses = () => ({
  type: 'GET_COURSES',
  payload: CourseService.getCourses()   // Promise — redux-promise resolves it
});
```

### Why `redux-thunk` is still in package.json

It was installed but the architecture was already committed to the promise/saga pattern before it was used. The package is unused and flagged for removal in `docs/plans/active/2026-06-26-package-audit.md`.

### Rule for future agents

Do not enable `thunk: true` in the store config. Do not write thunk-style action creators. All async actions use the `redux-promise` pattern (Promise payload) or `redux-saga` for side effects.

## Domain: Bugfix — PostgREST Function Overload Ambiguity (0 contacts + empty country dropdown)

### Context

After deploying the dual location filter, the admin tools messaging section showed 0 contacts and "No data" in the country dropdowns. The diagnostic query confirmed the 23-param `get_contact_segment` overload was deployed (pronargs = 23 exists alongside 16 and 17).

### Root cause

PostgreSQL allows multiple overloads of the same function. When the frontend sent the payload with only legacy params (the new params were null → stripped by the reducer), PostgREST received `{p_locationtype, p_matchallcourses, p_limit, p_offset}` and could not determine which `GetContactSegment` overload to call (old 15/17-param or new 23-param both accept those param names). PostgreSQL threw an ambiguity error; the service caught it and returned `[]`.

The empty country dropdown was a downstream consequence: with `audienceRows = []`, country options are derived from metadata (separate API call). That part was working — the dropdown was empty only because the initial segment call failed before any rows were available to populate it.

### Fix

`src/lob/AudienceMessaging.js` — `buildContactSegmentPayload()`:
- Changed `p_residency_exclude` from `null` (when no country) to always `Boolean(filters.residencyExclude)`
- Changed `p_birth_exclude` from `null` (when no country) to always `Boolean(filters.birthExclude)`

`false` is not stripped by the payload reducer. Sending `p_residency_exclude: false` includes a named parameter that only exists in the 23-param overload — PostgREST resolves to it unambiguously. The SQL logic is unaffected because `has_new_location_filters` only activates on non-null country or region, not the exclude flags.

### Recommended warehouse follow-up (not yet done)

Drop the old `TitulinoApi_v1.GetContactSegment` and `TitulinoApi_v1.GetContactSegmentCount` overloads (the pre-23-param ones). Run in Supabase to identify them:

```sql
SELECT proname, pronargs, pg_get_function_identity_arguments(p.oid)
FROM pg_proc p
JOIN pg_namespace n ON n.oid = p.pronamespace
WHERE n.nspname = 'TitulinoApi_v1'
AND p.proname IN ('GetContactSegment', 'GetContactSegmentCount')
ORDER BY proname, pronargs;
```

Then create a `27_drop_old_get_contact_segment_overloads.sql` migration that drops each non-23-param signature using `DROP FUNCTION IF EXISTS "TitulinoApi_v1"."GetContactSegment"(text, int, int, ...)` with the exact old signature.
