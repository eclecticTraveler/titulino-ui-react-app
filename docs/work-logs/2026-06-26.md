## Domain: Redux Middleware Architecture ŌĆö Why thunk is disabled

### Context

During the package audit, `redux-thunk` was flagged as unused. The store explicitly disables it. Documented here for future reference so any agent or developer understands the async middleware choice and does not re-enable thunk by accident.

### How the store middleware is configured

File: `src/redux/store/index.js`

```js
const customMiddlewares = [
  sagaMiddleware,
  reduxPromise,
  axiosMiddleware(HttpService.getAxiosClient()),
];

const store = configureStore({
  reducer: reducers,
  middleware: (getDefaultMiddleware) =>
    getDefaultMiddleware({ thunk: false, serializableCheck: false, immutableCheck: false })
      .concat(customMiddlewares),
});
```

RTK's `configureStore` includes `redux-thunk` by default. `{ thunk: false }` removes it from the defaults before the custom middleware is appended.

### What thunk does (and why this app doesn't need it)

Thunk lets an action creator return a **function** instead of a plain object. That function receives `dispatch` and does async work before dispatching the real action:

```js
// Thunk pattern ŌĆö NOT used here
const loadUser = () => async (dispatch) => {
  const data = await api.getUser();
  dispatch({ type: 'SET_USER', payload: data });
};
```

### What this app uses instead

| Middleware | Purpose |
|---|---|
| `redux-promise` | Action creators return a plain object with a **Promise as the payload**. The middleware waits for the Promise to resolve and dispatches the result automatically. |
| `redux-saga` | Handles complex async side effects (auth flows: SIGNIN, SIGNOUT, SIGNUP) via generator functions in `src/redux/sagas/`. |
| `redux-axios-middleware` | Wraps axios calls dispatched as actions ŌĆö automatically handles request/success/error lifecycle. |

The app's async action creator pattern (using `redux-promise`):

```js
// Promise pattern ŌĆö what this app uses
export const getCourses = () => ({
  type: 'GET_COURSES',
  payload: CourseService.getCourses()   // Promise ŌĆö redux-promise resolves it
});
```

### Why `redux-thunk` is still in package.json

It was installed but the architecture was already committed to the promise/saga pattern before it was used. The package is unused and flagged for removal in `docs/plans/active/2026-06-26-package-audit.md`.

### Rule for future agents

Do not enable `thunk: true` in the store config. Do not write thunk-style action creators. All async actions use the `redux-promise` pattern (Promise payload) or `redux-saga` for side effects.

## Domain: Bugfix ŌĆö PostgREST Function Overload Ambiguity + Invalid Column References

### Context

After deploying the dual location filter (v0.1.220), the admin tools messaging section showed 0 contacts and "No data" in the country dropdowns on dev. Two separate bugs were identified and resolved.

---

### Bug 1 ŌĆö PostgREST overload ambiguity (0 contacts on all payloads)

**Root cause:** `CREATE OR REPLACE FUNCTION` with a different parameter count creates a new overload ŌĆö it does not replace the old one. The old 17-param `GetContactSegment` and 15-param `GetContactSegmentCount` wrappers still existed alongside the new 23-param versions. When the frontend sent a payload with only legacy params (`p_locationtype`, `p_matchallcourses`, `p_limit`, `p_offset`), PostgREST could not determine which overload to call. PostgreSQL threw an ambiguity error; the service caught it and returned `[]`.

**Temporary JS workaround (applied, then reverted):** `buildContactSegmentPayload()` was changed to always send `p_residency_exclude: false` (never null) so the payload uniquely matched the 23-param signature. Committed as workaround, then reverted after the SQL fix was in place.

**Permanent SQL fix:** Two DROP migrations created and committed to titulino-warehouse (`645f06f`):
- `TitulinoApi_v1/2026/06/26_remove_old_get_contact_segment_wrapper_overload.sql` ŌĆö drops old 17-param `GetContactSegment`
- `TitulinoApi_v1/2026/06/26_remove_old_get_contact_segment_count_wrapper_overload.sql` ŌĆö drops old 15-param `GetContactSegmentCount`

**Frontend state after revert:** `p_residency_exclude` and `p_birth_exclude` send `null` when no country is selected (correct semantic). The JS file is clean ŌĆö no workaround code remains.

---

### Bug 2 ŌĆö Invalid column names in `Enrollment.get_contact_segment` (0 contacts when function runs)

**Root cause:** The Codex-generated `26_extend_get_contact_segment_dual_location_filters.sql` referenced four columns that do not exist in the `ContactResidencyLocation` / `ContactBirthLocation` composite types:
- `CountryDivisionId_Residency` ŌĆö does not exist
- `CountryDivisionResidencyNativeName` ŌĆö does not exist
- `CountryDivisionId_Birth` ŌĆö does not exist
- `CountryDivisionBirthNativeName` ŌĆö does not exist

The valid column names are `CountryDivisionResidencyName` (residency) and `CountryDivisionBirthName` (birth).

**Fix migration (created, pending commit approval):**
- `Enrollment/2026/06/26_fix_get_contact_segment_division_column_refs.sql` ŌĆö full `CREATE OR REPLACE FUNCTION` of the 23-param engine using only confirmed-valid column names
- Revert and verify files created alongside it

**Confirmed result after fix deployed:** 1370 contacts load on initial page open. Country dropdowns (Residency + Birth) populate from loaded contact rows. Include/Exclude toggles function independently per filter row.
